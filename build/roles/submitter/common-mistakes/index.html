Common Submitter Mistakes

Learn from the most frequent errors to improve your submission quality.

1. Vague or Ambiguous Intents​

Problem​

Intents that don't clearly define success or allow multiple interpretations.

✅ Good Example

Intent: "Provide a factual, cited answer to a specific historical question using web search"

Clear success criteria and scope

❌ Bad Example

Intent: "Help the user with their question"

Too vague - what kind of question? What does "help" mean?

2. System Prompts That Don't Enable the Intent​

Problem​

System prompt doesn't give the AI the tools or instructions needed to fulfill the intent.

✅ Good Example

Intent: Search the web for recent news

System Prompt: "You are a helpful assistant with access to web search tools. When users ask about recent events, use the news_search tool to find current information."

Includes tool permissions and usage guidance

❌ Bad Example

Intent: Search the web for recent news

System Prompt: "You are a helpful assistant."

Missing tool permissions and search instructions

3. Unrealistic Failure Modes​

Problem​

Bad responses that are obviously terrible or wouldn't realistically come from an AI.

Common Error

Creating bad responses that are completely nonsensical or include obvious errors like:

  • Random gibberish
  • "ERROR: System malfunction"
  • Completely off-topic responses

These don't represent real failure modes!

4. Good and Bad Responses Too Similar​

Problem​

The difference between good and bad responses isn't clear or meaningful.

Best Practice

The contrast should be obvious to reviewers. If you have to explain the difference in detail, the examples probably aren't distinct enough.

5. Missing Context in User Prompts​

Problem​

User prompts that assume information not provided in the conversation.

See Example

Bad: "What happened next?" (Reader doesn't know what "next" refers to)

Good: "After the Apollo 11 landing in 1969, what was the next major milestone in the space program?"

6. Over-Complicated Scenarios​

Problem​

Creating unnecessarily complex edge cases that test multiple things at once.

Keep It Simple

Focus on one clear failure mode per submission. Complex scenarios make it harder for reviewers and less useful for training.
